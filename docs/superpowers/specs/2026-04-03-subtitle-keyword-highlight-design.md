# 字幕关键词高亮设计

## 背景

当前项目的字幕渲染能力较基础，只支持字号、颜色和位置，字幕整体以单个文本节点输出，适合普通口播字幕，但不足以承载更强的观点表达。

本轮目标不是把字幕做成综艺感或短视频爆字风，而是为 16:9 视频博客建立一套稳定、可持续使用的“关键词高亮”能力：

- 适合长视频持续观看
- 能突出关键观点、结论词和数字信息
- 与现有项目的深色、科技感、紫色强调视觉体系保持一致
- 允许通过 AI 自动判断高亮词，而不是在前端写死高亮规则

## 结论

采用 `AI 决策 + 前端渲染` 的方案：

- AI 负责判断每条字幕是否需要高亮，以及需要高亮的精确文本区间
- 前端不参与语义判断，只负责校验和渲染
- 高亮视觉采用 `Impact Highlight` 风格，即白字主句 + 单个关键词高亮块 + 底部安全区 + 轻量强调动画

第一阶段仅支持：

- 每条字幕最多高亮 1 段
- 关键词高亮
- 16:9 横版视频优先

第一阶段不支持：

- 多段高亮
- 逐词高亮
- 用户自定义 prompt
- AI 决定视觉样式

## 现状诊断

### 1. 当前字幕结构缺少高亮语义层

现有 `SrtEntry` 只包含：

- `index`
- `startMs`
- `endMs`
- `text`

说明当前字幕数据只表达“时间 + 文本”，没有附加语义层，无法直接支持某个词的独立视觉表达。

### 2. 当前字幕渲染为整句单节点输出

现有字幕渲染组件按当前时间找到一条字幕后，直接将整段文本输出为一个 `span`。这意味着如果要支持关键词高亮，渲染层必须从“整句输出”改为“前缀 + 高亮段 + 后缀”的分段输出。

### 3. 项目整体视觉方向已形成稳定语言

当前编辑器与 AI 卡片设计已经形成以下视觉倾向：

- 深色背景
- 高对比白字
- 紫色作为核心强调色
- 适度发光、阴影与玻璃质感

因此字幕不应采用强综艺、卡通、花哨描边路线，而应保持专业与冲击力之间的平衡。

## 设计目标

### 1. 强调观点，而不是抢走画面

字幕需要帮助用户快速抓住观点，但不能持续压制画面主体。高亮只能作为局部强调，而不是整个字幕持续高对比噪声。

### 2. 语义判断和视觉渲染解耦

AI 的职责是“圈出哪一段值得高亮”，而不是“决定怎么展示”。这保证了：

- 模型输出可控
- 视觉风格统一
- 后续替换模型时不会影响渲染层

### 3. 对编辑器用户保持可理解、可回退

用户应理解：

- AI 是在生成“关键词标注”
- 样式是统一控制的
- 结果过期后可以重新生成

## 风格方案：Impact Highlight

### 视觉表现

字幕整体采用底部居中的两行布局，主文本为高权重白字。被标记为关键词的片段渲染为带圆角的高亮块。

建议默认视觉参数：

- 位置：底部安全区
- 底边偏移：`96px`
- 字号：`56px`
- 字重：`800`
- 行高：`1.3`
- 最大行数：`2`
- 主文字色：`#F8FAFC`
- 高亮底色：`#F8DC48`
- 高亮文字色：`#111827`

### 动效策略

整句字幕采用轻微淡入与上浮，避免重节奏跳动。高亮词额外获得一次短促的强调动效，例如：

- 轻微缩放
- 轻微擦入

高亮不应使用持续闪烁、频繁弹跳或整句放大等强刺激表现。

### 使用边界

- 一句字幕最多高亮 1 个片段
- 没有明显重点时允许完全不高亮
- 高亮块长度应保持短小，避免覆盖整句

## AI 职责与输出约束

### AI 负责什么

AI 负责根据当前字幕文本与上下文，判断：

- 是否应该高亮
- 哪个片段最值得高亮
- 该片段在原文本中的精确区间

### AI 不负责什么

AI 不负责：

- 颜色
- 字号
- 位置
- 动效
- 多个可选方案排序

这些均由系统与前端统一控制。

### 输出结构

AI 返回的最小结构为：

```json
{
  "entryIndex": 12,
  "shouldHighlight": true,
  "highlightText": "世界冠军",
  "start": 4,
  "end": 8
}
```

约束如下：

- `start` / `end` 使用 JavaScript 字符串下标
- `end` 为 `exclusive`
- `highlightText` 必须严格等于 `sourceText.slice(start, end)`
- `shouldHighlight = false` 时：
  - `highlightText = ""`
  - `start = -1`
  - `end = -1`

### AI 判定标准

系统给模型的规则应明确说明：

- 优先高亮：结论词、数字、身份词、首次/突破词、强反差词
- 避免高亮：虚词、口头语、模糊词、整句长短语
- 一条字幕只选一个最强信息点
- 如果没有明确重点，应返回不高亮

建议同时提供当前字幕前后 1 到 2 条上下文，避免模型仅凭单句误判。

## 数据模型设计

### 原始字幕与高亮结果分离

保留原始 `SrtEntry` 结构不变。新增独立的高亮结果结构，用于表示 AI 标注结果。

建议新增：

```ts
interface SubtitleHighlight {
  entryIndex: number;
  start: number;
  end: number;
  highlightText: string;
  sourceText: string;
}
```

其中：

- `entryIndex` 用于定位字幕条目
- `sourceText` 用于校验结果是否仍与当前字幕文本一致

### 时间线数据挂载方式

建议在 `TimelineData` 中新增 `subtitleHighlights`，而不是把高亮结果写回 `SrtEntry`。

原因：

- 原始字幕保持纯净
- AI 标注属于衍生数据
- 字幕文本变更后可自然失效
- 后续批量重算更容易

## 校验与失效规则

所有 AI 返回结果在进入系统前都必须经过严格校验。

### 基础校验

- `entryIndex` 必须能定位到现有字幕
- `start >= 0`
- `end > start`
- `end <= sourceText.length`
- `highlightText === sourceText.slice(start, end)`

### 过期校验

持久化时保存 `sourceText`。在渲染或编辑时，如果当前字幕文本与 `sourceText` 不一致，则说明字幕内容已经变化，这条高亮结果自动视为过期。

过期结果：

- 不参与渲染
- UI 提示用户可重新生成高亮

### 回退机制

任何非法或过期高亮都不应影响字幕显示。系统必须回退为普通字幕渲染。

## 编辑器设置项设计

第一阶段仅暴露真正必要的视觉与操作项。

### 建议开放

- `启用关键词高亮`
- `高亮底色`
- `高亮文字颜色`
- `高亮圆角`
- `高亮内边距`
- `高亮动画`
- `生成关键词高亮`
- `重新生成高亮`

### 暂不开放

- prompt 编辑
- 多高亮段数量控制
- 高亮长度阈值
- 模型温度等模型参数
- 坐标模式切换
- 多套复杂风格系统

这些能力会增加理解成本，应等基础体验稳定后再考虑开放。

## AI 触发时机

第一阶段采用 `显式触发 + 过期提示` 策略。

### 触发入口

- 导入 SRT 后，用户手动点击“生成关键词高亮”
- 字幕文本编辑后，提示重新生成
- 用户主动点击“重新生成高亮”

### 不触发 AI 的场景

- 仅修改视觉样式参数时
- 仅切换主题色或视频整体风格时

这些场景只需要重渲染，不需要重新做语义判断。

## 渲染方案

渲染层根据当前字幕条目查找是否存在合法高亮标注。

如果存在，则将文本拆为三段：

- `before`
- `highlight`
- `after`

分别渲染为：

- 普通白字
- 高亮块
- 普通白字

如果不存在合法高亮，则回退为普通整句字幕。

这种方案兼顾：

- 视觉连续性
- 简单实现
- 可预测性

## 错误处理

### AI 调用失败

当模型请求失败、超时或返回非法 JSON 时：

- 保持字幕轨可正常使用
- 不写入任何高亮结果
- 在 UI 中提示用户检查 AI 配置后重试

### 部分结果失败

当某一批字幕中只有部分结果合法时：

- 合法结果照常写入
- 非法结果忽略
- 不应因个别条目失败导致整批丢弃

## 非目标

本设计暂不覆盖：

- 逐词高亮
- 字幕逐字动画
- 多段同时高亮
- 竖版 9:16 专用布局优化
- 用户自定义高亮 prompt 模板
- AI 自动判断不同视觉风格

## 验收标准

当以下条件满足时，说明本设计落地成功：

- 用户可为字幕轨启用关键词高亮
- AI 可为字幕批量输出结构化高亮区间
- 非法结果不会破坏字幕渲染
- 字幕文本修改后，高亮结果可自动失效
- 用户可手动重新生成高亮
- 16:9 视频中高亮字幕具备清晰且不过度抢画面的表现
